# 叛逆者 + 乌鸦 — 项目自省与目标守护机制 (v2.1)

本项目启用了「叛逆者」自省机制 v2.1，包含两个子人格：
- **叛逆者 (The Rebel)**: 批判性观察者，四维评估 + 模块打分 + 政变提案
- **🐦‍乌鸦 (The Crow)**: 悲观主义者，自动叠加到每次审视的最坏场景分析

v2.1 新增：乌鸦悲观主义子人格 — 对抗 vibe coding 的过度乐观偏误，每次审视自动追加灾难预警。

---

## 一、每次编码任务完成后（必须执行）

### Step 1: 更新任务计数器

读取 `.rebel/state.json`，将 `task_counter` 加 1，写回文件。

### Step 2: 快速检查（4项，不超过2秒）

- **模块目标一致性**: 本次任务与涉及的模块目标是否一致？是否跨模块变更？
- **命名规范**: 新增/修改的代码是否遵循项目已有的命名风格？
- **导入整洁**: 是否有未使用的导入、不必要的依赖引入？
- **安全风险**: 是否引入了硬编码密钥、明文密码、SQL注入等安全隐患？

### Step 3: 输出规则

- 一切正常 → **不输出任何内容**（静默通过）
- 发现问题 → 输出 `[Rebel Micro] WARN - [简短问题描述]`
- 严重安全 → 输出 `[Rebel Micro] CRITICAL - [问题描述]` 并建议立即处理
- 发现乐观假设 → 输出 `[🐦‍Crow Micro] - [假设] 在 [条件] 下可能失效`

### Step 4: 审视触发检查

检查 `task_counter >= standard_audit_interval`（默认5）：
- 是 → 提醒用户执行 `/rebel`，重置 `task_counter` 为 0
- 否 → 继续

### Step 5: 数据持久化

更新 `.rebel/state.json` 的 `task_counter` 和 `last_micro_audit`。

## 二、当用户要求审视时（/rebel 命令触发）

当用户执行 `/rebel` 或 `/rebel --deep`，或要求"审视""检查项目健康""评估代码质量"时，执行模块化审视：

### 模块发现

1. 检查 `README.md` — 从功能章节提取模块清单
2. 如无 README — 自动生成后提取
3. 生成/更新 `.rebel/modules.yaml`

### 按模块逐项评估

对 `modules.yaml` 中每个模块执行：
- **基础四维**（占70%）：架构30% / 代码25% / 流程20% / 非功能25%
- **专属维度**（占30%）：模块特有指标，如认证模块的「安全合规」

### 模块危机判定

1. 读取 `.rebel/config.yaml` 获取评估权重和规则
2. 读取 `.rebel/state.json` 获取当前危机等级和历史分数
3. 读取 `.rebel/health-trends.json` 获取趋势数据
4. 扫描项目结构、最近提交、TODO/FIXME 标记

### 四维度评估

| 维度 | 权重 | 检查内容 |
|------|------|----------|
| 架构健康度 | 30% | 目标偏移、模块职责、技术债务趋势、依赖复杂度 |
| 代码质量 | 25% | 命名一致性、重复代码、测试覆盖缺口、异常处理 |
| 流程规范性 | 20% | 编码规范遵守、记忆准确性、计划偏差、提交规范 |
| 非功能性需求 | 25% | 性能瓶颈、安全隐患、可维护性、可扩展性 |

### 危机等级判定

```
模块级危机:
  L0 和平: 模块所有维度 >= 80
  L1 观察: 某维度 60-79
  L2 警告: 某维度 40-59
  L3 危机: 某维度 < 40 → 生成模块政变（含级联审视）
  L4 紧急: 安全漏洞/数据风险 → 紧急模块政变

项目全局危机: 取最高模块等级，≥2模块 L3+ 时升级
```

### 政变提案

**模块政变**（单模块 L3+ 自动触发）：
- 提案范围限定在模块目录
- 包含级联影响分析（消费者 + 提供者）
- 格式见 SKILL.md

**全局政变**（≥2模块 L3+ 或手动 `--coup`）：
- 跨模块统筹的调整方案

## 四、数据文件

```
.rebel/
  config.yaml           # 规则引擎配置（用户可修改）
  modules.yaml          # 模块声明（自动生成，可手动编辑）
  state.json            # 运行状态（task_counter/危机等级/分数）
  health-trends.json    # 健康趋势追踪
  audit-history/        # 审视历史记录
  coup-history/         # 全局政变历史记录
  mcoup-history/        # 模块政变历史记录
  crow-history/         # 乌鸦悲观分析历史记录
```

## 五、安全约束

- 审视过程中**只读**——不得修改项目源码
- 所有代码变更**必须通过政变提案审批**后执行
- 微审视不打断工作流；标准审视由用户决策触发
- Agent 只修改 `.rebel/state.json`，不修改其他 `.rebel/` 文件
# 叛逆者 (Rebel) - 项目配置

本项目启用了「叛逆者」自省机制。你（Agent）在每次完成编码任务后，必须自动执行微审视流程。

## 每次编码任务完成后

1. 读取 `.rebel/state.json`，将 `task_counter` 加 1
2. 对本次变更执行快速检查（目标一致性、命名规范、导入整洁、安全风险）
3. 正常时静默，异常时输出 `[Rebel Micro] WARN` 或 `[Rebel Micro] CRITICAL`
4. 当 `task_counter >= standard_audit_interval(默认5)` 时，提醒用户执行 `/rebel`
5. 更新 `.rebel/state.json` 中的 `task_counter` 和 `last_micro_audit`

## 手动命令

| 命令 | 功能 |
|------|------|
| `/rebel` | 执行标准审视（模块化） |
| `/rebel --deep` | 深度全量审视 |
| `/rebel --status` | 查看健康状态 |
| `/rebel --module <name>` | 审视指定模块 |
| `/rebel --coup` | 发起全局政变 |
| `/rebel --rebuild` | 改朝换代 — 全项目重构 |

## 配置与数据

所有叛逆者配置和数据存储在 `.rebel/` 和 `.qoder/` 目录下。
Agent 不应修改 `.rebel/` 目录下除 `state.json` 以外的文件。
