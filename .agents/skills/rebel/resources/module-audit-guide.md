# 模块级审视操作指南 (Module Audit Guide)

本文件定义了叛逆者 v2.0 模块化审视的详细操作流程。

## 一、模块发现流程

### 1.1 从 README 提取模块

叛逆者自动解析 README.md 的功能章节来识别模块边界：

**支持的 README 结构模式**：

```markdown
## 功能 (Features)

### 用户认证
- 注册/登录
- JWT Token 管理
...

### 数据管理
- CRUD 操作
- 数据导出
...
```

```markdown
## Features

### Authentication
- User login/register
...

### Data Management
...
```

**解析规则**：
- 匹配 `## 功能`、`## Features`、`## 模块` 等二级标题
- 其下三级标题为模块名
- 提取下方的功能描述为模块说明
- 自动关联模块到对应的代码目录（扫描目录结构匹配模块名）

### 1.2 无 README 时自动生成

当项目没有 README.md 时，自动生成流程：

1. **提取项目目标**
   - 从 `AGENTS.md` 读取项目描述
   - 从 `package.json`/`requirements.txt` 等读取项目名称和依赖

2. **扫描代码结构**
   - 遍历项目根目录，识别每个子目录
   - 排除 `.git`、`node_modules`、`venv` 等通用目录
   - 识别入口文件（`index.*`、`main.*`、`app.*`）

3. **归纳功能模块**
   - 根据目录名推演模块名称
   - 根据目录内的代码文件推断模块功能
   - 生成 README 初稿

4. **输出 README.md**
   - 用户确认后写入项目根目录
   - 包含：项目名称、简介、功能列表、技术栈、快速开始

### 1.3 生成 modules.yaml

模块声明文件格式 (`.rebel/modules.yaml`)：

```yaml
# 模块声明 — 由叛逆者自动生成，可手动编辑
# 最后更新: 2026-06-25T16:30:00

modules:
  - id: "user-auth"
    name: "用户认证模块"
    path: "src/auth/"
    description: "用户注册、登录、Token管理、权限控制"
    goal: "提供安全的用户身份认证和会话管理"
    weight: 0.20
    dependencies: ["data-storage"]     # 本模块依赖的模块
    depended_by: ["api-gateway"]       # 依赖本模块的模块
    extra_dimensions:
      - name: "security_compliance"
        label: "安全合规"
        weight: 1.0
        indicators:
          - "密码加密强度"
          - "Token刷新机制"
          - "暴力破解防护"
```

## 二、模块评分指南

### 2.1 基础四维评估

每个模块按标准四维评估，但聚焦于**模块自身范围内**：

| 维度 | 模块内检查重点 | 区别 |
|------|--------------|------|
| 架构健康度 | 模块内目标偏移、模块职责内聚、模块内TODO密度、模块内依赖复杂度 | 聚焦模块作用域 |
| 代码质量 | 模块内命名一致性、模块内重复代码、模块测试覆盖、模块内异常处理 | 仅检查模块目录 |
| 流程规范性 | 模块内编码规范、模块内记忆准确性、模块内变更计划偏差、模块内提交规范 | 模块级粒度 |
| 非功能性需求 | 模块内性能、模块内安全、模块内可维护性、模块内可扩展性 | 模块级风险 |

### 2.2 专属维度评估

每个模块可追加 1-3 个专属维度。专属维度独立于基础四维。

**常见专属维度参考**：

| 模块类型 | 建议的专属维度 |
|----------|---------------|
| 认证模块 | security_compliance（安全合规）— 加密强度、会话管理、暴力破解防护 |
| API网关 | reliability（可靠性）— 限流机制、熔断策略、错误恢复 |
| 数据存储 | data_integrity（数据完整性）— 事务处理、一致性、备份 |
| 前端UI | responsiveness（响应式与可访问性）— 性能指标、无障碍支持 |
| 消息队列 | message_reliability（消息可靠性）— 投递保证、重试策略、幂等 |
| 日志/监控 | observability（可观测性）— 日志级别、指标暴露、告警 |

**专属维度评分标准**：

- 90-100: 所有指标达标，有完善的最佳实践
- 70-89: 大部分指标达标，有小幅改进空间
- 50-69: 部分指标缺失或不足，需要关注
- 30-49: 明显不足，影响功能质量
- <30: 严重缺陷，影响系统安全和稳定性

### 2.3 模块分计算公式

```
无专属维度:
  模块分 = (架构×0.30 + 代码×0.25 + 流程×0.20 + 非功能×0.25)

有专属维度:
  基础分 = (架构×0.30 + 代码×0.25 + 流程×0.20 + 非功能×0.25)
  专属分 = Σ(专属维度i × weight系数i) / Σweight系数i
  模块分 = 基础分 × 0.70 + 专属分 × 0.30
```

## 三、模块级政变指南

### 3.1 触发条件

| 条件 | 级别 | 行为 |
|------|------|------|
| 任一维度 < 40 | L3 | 生成模块政变提案 |
| 专属维度 < 30 | L3 | 生成模块政变提案 |
| 安全漏洞 | L4 | 生成紧急模块政变提案 |
| ≥2模块同时 L3+ | — | 触发全局政变 |

### 3.2 级联审视

触发模块政变时，自动执行级联审视：

```
Step 1: 确定影响范围
  - 从 modules.yaml 读取 depended_by（消费者）
  - 从 modules.yaml 读取 dependencies（提供者）

Step 2: 对消费者模块快速审视
  - 检查：政变是否导致消费者接口不兼容？
  - 评估：消费者是否需要适配变更？
  - 输出：影响等级 (high/medium/low) + 建议操作

Step 3: 对提供者模块快速审视
  - 检查：政变是否需要提供者配合变更？
  - 输出：影响等级 + 建议操作

Step 4: 汇总级联影响报告
  - 总受影响的模块数
  - 各模块影响等级和建议操作
```

### 3.3 模块政变 vs 全局政变

| 维度 | 模块政变 | 全局政变 |
|------|---------|---------|
| 触发 | 单模块 L3+ | ≥2模块 L3+ 或手动 |
| 范围 | 单一模块目录 | 全项目 |
| 提案数 | 1-3 条 | 可多条跨模块 |
| 级联 | 自动级联审视 | 可能包含多个模块的提案 |
| 执行 | 逐模块执行 | 统筹执行 |

## 四、报告输出规范

### 4.1 模块状态着色

- 🟢 绿色: 模块分 >= 70（健康）
- 🟡 黄色: 模块分 40-69（警告）
- 🔴 红色: 模块分 < 40（危机）

### 4.2 瓶颈分析格式

当「木桶短板分」低于「加权平均分」10分以上时，专门输出瓶颈分析：

```
───────────────────────────────────────────────
  瓶颈分析
───────────────────────────────────────────────

  🔴 [瓶颈模块名] (63/100) — 拉低项目整体健康度 [N] 分

  拖累因素:
  ├── 代码质量 (55): 模块内发现3处重复代码块
  │   → 文件: src/storage/cache.ts:45-78, 120-153
  └── 架构健康度 (60): 模块与 data-storage 存在循环依赖
      → 问题: src/storage/ 同时被 3 个模块直接引用

  改进建议: [具体建议]
```

### 4.3 趋势对比

模块级趋势对比显示每个模块的分数变化：

```
  模块趋势:
  ├── user-auth: 72 → 75 ↑ (安全合规优化后)
  ├── api-gateway: 85 → 85 → (稳定)
  ├── data-storage: 68 → 63 ↓ (新增代码质量下降)
  └── notification: 42 → 39 ↓ (触发政变)
```

## 五、与改朝换代的联动

`/rebel --rebuild` 的 Phase 1 Step 3 功能累积审计会直接引用模块审视结果：

```
功能审计:
  user-auth (最近审视: 72/100)
    → 分类: ✅ 保留
    → 理由: 模块健康，无重大问题
  
  data-storage (最近审视: 63/100)
    → 分类: 🔄 重构
    → 理由: 代码质量持续下降，循环依赖问题未解决
  
  notification (最近审视: 39/100, L3危机)
    → 分类: 🔄 重构（优先级最高）
    → 理由: 模块已触发政变，架构需要根本性调整
```
