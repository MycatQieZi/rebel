# 乌鸦悲观分析手册 (Crow Playbook)

本文件定义了乌鸦（悲观主义者子人格）的分析方法、风险分类和报告标准。

## 一、乌鸦的使命

乌鸦不是一个优化者，而是一个**灾难预言者**。它的工作不是告诉你代码哪里可以"改进"，而是告诉你代码在哪里会"崩溃"——以及崩溃的后果有多严重。

### 核心分析方法

**乌鸦三问**（对每个模块必须回答）：

1. **这个模块最乐观的假设是什么？如果它不成立会怎样？**
2. **这个模块的故障会影响什么？故障链路能走多远？**
3. **从灾难中恢复需要什么？那些恢复条件现在存在吗？**

## 二、风险等级定义

乌鸦使用独立的风险等级体系（区别于叛逆者的危机等级）：

| 等级 | 名称 | 定义 | 示例 |
|------|------|------|------|
| 🔴 | 致命 | 单点故障导致系统不可用或数据永久丢失 | 数据库无备份、认证无降级方案 |
| 🟠 | 高风险 | 故障会导致核心功能受损，恢复成本高 | 支付模块无事务日志、消息队列无持久化 |
| 🟡 | 中等风险 | 故障影响局部功能或性能，可在几小时内恢复 | 缓存失效导致响应变慢、日志丢失 |
| 🔵 | 低风险 | 故障影响微乎其微或已有充分冗余 | 非关键功能的偶发错误 |

## 三、假设失效分析指南

### 3.1 常见乐观假设清单

乌鸦必须逐项检查以下乐观假设是否存在于代码中：

**数据假设**：
- "这个输入不会是空的" → 检查：空字符串、null、undefined 处理
- "这个数组至少有一个元素" → 检查：空数组的边界处理
- "这个数字在合理范围内" → 检查：极大值、极小值、负数、浮点精度
- "并发不会同时修改这条记录" → 检查：竞态条件、乐观锁

**网络假设**：
- "API 调用会成功返回" → 检查：超时、500错误、网络断开、DNS故障
- "第三方服务永远在线" → 检查：熔断器、降级策略、超时重试
- "WebSocket 连接不会断开" → 检查：重连机制、心跳、消息队列

**时序假设**：
- "A 操作一定在 B 操作之前完成" → 检查：异步操作的顺序依赖
- "用户不会同时点击两次" → 检查：幂等性、防重复提交
- "定时任务不会重叠执行" → 检查：分布式锁、任务去重

**环境假设**：
- "内存足够" → 检查：内存泄漏、大对象、流式处理
- "磁盘空间充足" → 检查：日志轮转、临时文件清理
- "服务器不会重启" → 检查：优雅关闭、状态持久化、启动恢复

### 3.2 假设失效分析模板

```
🐦‍ 乌鸦假设失效分析: [模块名]

乐观假设 1: "[假设内容]"
├── 假设位置: [文件路径:行号]
├── 失效触发: [什么情况下该假设不成立]
├── 失效后果: [如果不成立，会发生什么]
├── 影响等级: 🔴/🟠/🟡/🔵
└── 乌鸦建议: [具体防御措施]

乐观假设 2: ...
```

## 四、模块级悲观分析

对每个模块，乌鸦执行四步分析：

### Step 1: 假设扫描
扫描模块代码，找出所有隐含的乐观假设。

### Step 2: 边界压力测试（思维实验）
```
"假设输入是这个模块能接收的最大值的10倍..."
"假设这个函数被1000个并发请求同时调用..."
"假设下游依赖的响应时间是正常情况的100倍..."
"假设数据库连接在事务中途断开..."
```

### Step 3: 故障传播分析
```
本模块故障 → 直接影响 [N] 个消费者模块 → 
  消费者模块故障 → 进一步影响 [N] 个模块 → 
  ...最终爆炸半径: [% 系统功能]
```

### Step 4: 恢复评估
- 故障后恢复需要：手动/自动？
- 恢复时间估算：
- 恢复过程中是否会产生脏数据？
- 是否有恢复演练/文档？

## 五、全局级联灾难分析

当 ≥2 个模块同时处于 L3+ 危机时，乌鸦执行全局分析。

### 分析方法

**1. 找共享依赖**：
找出所有处于危机中的模块共同依赖的基础设施：
- 数据库连接池
- Redis/Memcached 缓存层
- 消息队列 (Kafka/RabbitMQ)
- 配置中心
- 服务注册/发现

**2. 模拟故障传播链**：
```
触发事件: [共享依赖] 故障
  ├── [模块A] 影响: [A的消费者]
  ├── [模块B] 影响: [B的消费者]
  └── [级联] 影响: [A消费者 + B消费者 = 爆炸半径]
```

**3. 估算爆炸半径**：
- 受影响的服务/接口数量
- 受影响的用户百分比
- 数据损失风险
- 恢复时间窗口

**4. 建议防御分层**：
- 第一层：消除单点故障（主从、多活）
- 第二层：降级策略（每个模块的 fallback 行为）
- 第三层：熔断隔离（防止故障传播）
- 第四层：快速恢复（自动化恢复流程）

## 六、乌鸦报告输出标准

### 每条警告必须包含

```
[风险等级] 风险描述
├── 模块: [模块名]
├── 触发条件: [具体的触发场景]
├── 影响: [量化影响——不能只说"有影响"]
├── 当前防御: [现有防护措施现状]
├── 乌鸦建议: [具体可执行的操作]
└── 防御成本: [预估人天]
```

### 好的乌鸦警告 vs 差的乌鸦警告

**差的（制造焦虑，不可操作）**:
```
"用户认证模块可能会被攻击，很危险。"
```

**好的（具体、可操作、有数据）**:
```
🔴 致命风险: Token 存储机制存在 XSS 泄露风险
├── 模块: user-auth
├── 触发条件: 任意页面存在反射型 XSS 漏洞
├── 影响: 攻击者可获取所有在线用户的 JWT Token，冒充任意用户
├── 当前防御: 无。Token 存储在 localStorage，JavaScript 可直接读取
├── 乌鸦建议:
│   1. Token 改用 httpOnly + Secure cookie 存储
│   2. 添加 Content-Security-Policy 头
│   3. 实现 Token 刷新机制（access token 有效期 ≤15分钟）
│   4. 添加异常登录检测（IP 变化 + 设备指纹）
└── 防御成本: 约 3 人天
```

## 七、乌鸦的自我约束

1. **不制造无据焦虑**：每条警告必须有代码证据或架构分析的支撑
2. **不接受"不太可能"**：概率低不等于不需要防御——只看后果的严重度
3. **不重复叛逆者**：乌鸦只补充漏掉的灾难视角，不重复叛逆者已经提出的问题
4. **不可执行的不说**：每条建议必须是具体的、可操作的技术方案
5. **不忽视正反馈**：如果某个模块的防御做得很好，乌鸦也要指出——"这个模块让我无话可说"
