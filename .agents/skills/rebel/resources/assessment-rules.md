# 叛逆者评估规则手册 (Assessment Rules)

本文件定义了叛逆者在审视过程中使用的详细评估规则和检查方法。

## 一、架构健康度评估规则

### 1.1 目标偏移度检测 (goal_drift)

**检查方法:**
1. 提取项目的核心目标声明（来自 README、PLAN、或项目初始配置）
2. 分析最近 N 次任务/提交的内容
3. 计算每个任务与核心目标的关联度（0-1）
4. 如果关联度 < 0.3 的任务占比超过 40%，判定为"目标偏移"

**评分标准:**
- 90-100: 所有近期任务与核心目标高度一致
- 70-89: 偶尔有偏离任务，但总体方向正确
- 50-69: 明显偏移，多个任务与核心目标无关
- 30-49: 严重偏移，大部分精力花在非核心事务
- 0-29: 完全迷失方向

**红旗信号:**
- 连续3次以上提交与核心目标无关
- 引入了与项目定位不符的大型依赖
- 项目描述与实际功能严重不符

### 1.2 模块职责清晰度 (module_cohesion)

**检查方法:**
1. 列出所有顶级模块/目录
2. 分析每个模块的文件数量、行数、职责数量
3. 检查跨模块依赖关系
4. 识别"上帝模块"（职责过多的模块）和"孤儿模块"（无引用的模块）

**评分标准:**
- 90-100: 模块边界清晰，职责单一，依赖合理
- 70-89: 基本清晰，少量模块职责边界模糊
- 50-69: 多个模块职责混乱，存在循环依赖
- 30-49: 严重耦合，模块边界名存实亡
- 0-29: 单体混乱架构

**红旗信号:**
- 单个文件超过 1000 行
- 模块之间存在循环依赖
- 一个模块承担了超过3个不相关的职责

### 1.3 技术债务趋势 (tech_debt_trend)

**检查方法:**
1. 统计 TODO、FIXME、HACK、XXX、WORKAROUND 标记数量
2. 与上次审视对比，计算增减趋势
3. 分析已知问题的修复进度
4. 检查是否有临时方案长期未替换

**评分标准:**
- 90-100: 债务数量稳定减少，无新增
- 70-89: 债务数量稳定，新增和修复平衡
- 50-69: 债务缓慢增长，新增多于修复
- 30-49: 债务快速增长，大量临时方案
- 0-29: 债务失控，代码充满 TODO 和 HACK

**红旗信号:**
- TODO/FIXME 数量比上次审视增加超过 50%
- 存在超过 30 天未处理的 FIXME
- 临时方案（HACK/WORKAROUND）数量超过正式方案

### 1.4 依赖复杂度 (dependency_complexity)

**检查方法:**
1. 统计直接依赖和间接依赖数量
2. 检查是否有未使用的依赖
3. 检查是否有多个依赖完成同一功能
4. 评估依赖的版本管理策略

**评分标准:**
- 90-100: 依赖精简，每个都有明确用途
- 70-89: 依赖合理，少量可优化
- 50-69: 依赖偏多，存在冗余
- 30-49: 依赖臃肿，维护困难
- 0-29: 依赖地狱

## 二、代码质量评估规则

### 2.1 命名一致性 (naming_consistency)

**检查方法:**
1. 扫描变量、函数、类的命名模式
2. 检查是否混用不同命名风格（camelCase vs snake_case vs kebab-case）
3. 检查是否有含义模糊的命名（如 a, b, temp, data）
4. 检查命名是否反映了实际用途

**评分标准:**
- 90-100: 命名风格完全一致，含义清晰
- 70-89: 基本一致，少量例外
- 50-69: 混用多种风格，部分命名含义模糊
- 30-49: 大量不一致，命名随意
- 0-29: 完全混乱

### 2.2 重复代码检测 (code_duplication)

**检查方法:**
1. 搜索相似的代码块（结构相似，变量名不同）
2. 检查复制粘贴的模式
3. 识别可以通过提取函数/类消除的重复

**评分标准:**
- 90-100: 几乎无重复代码
- 70-89: 少量重复，不影响维护
- 50-69: 明显重复，增加了维护成本
- 30-49: 大量重复，修改需要同步多处
- 0-29: 代码库充满复制粘贴

### 2.3 测试覆盖缺口 (test_coverage_gap)

**检查方法:**
1. 统计测试文件和测试用例数量
2. 识别关键业务逻辑是否有测试覆盖
3. 检查边界条件和异常路径的测试
4. 评估测试的质量和可靠性

**评分标准:**
- 90-100: 关键逻辑全覆盖，测试质量高
- 70-89: 大部分覆盖，少量缺口
- 50-69: 明显缺口，部分关键逻辑未测试
- 30-49: 测试严重不足
- 0-29: 几乎没有测试

### 2.4 异常处理 (error_handling)

**检查方法:**
1. 搜索空的 catch/except 块
2. 检查是否有静默失败（吞掉错误不处理）
3. 检查错误信息是否有意义
4. 评估错误恢复策略

**评分标准:**
- 90-100: 异常处理完善，错误信息清晰
- 70-89: 基本完善，少量遗漏
- 50-69: 部分处理，存在静默失败
- 30-49: 大量未处理异常
- 0-29: 完全没有异常处理

## 三、流程规范性评估规则

### 3.1 编码规范遵守率 (coding_standards)

**检查方法:**
1. 检查代码格式化（缩进、空格、括号风格）
2. 检查文件头注释和文档字符串
3. 检查导入语句的组织
4. 与项目的 style guide 对比

### 3.2 记忆准确性 (memory_accuracy)

**检查方法:**
1. 交叉引用项目记忆与实际代码
2. 检查记忆是否描述了已不存在的功能
3. 检查记忆中的技术栈是否与实际一致
4. 识别矛盾的记忆条目

### 3.3 计划执行偏差 (plan_deviation)

**检查方法:**
1. 对比原始计划与实际实现
2. 标记未完成的计划项
3. 标记计划外新增的功能
4. 评估偏差的合理性

### 3.4 提交规范 (commit_hygiene)

**检查方法:**
1. 分析 commit message 格式
2. 检查是否遵循约定式提交（Conventional Commits）
3. 检查提交粒度是否合理
4. 识别无意义的提交信息

---

## 四、非功能性需求评估规则

### 4.1 性能瓶颈 (performance_risk)

**检查方法:**
1. 搜索性能反模式（N+1 查询、内存泄漏模式、同步阻塞等）
2. 检查大文件处理和流式传输
3. 评估缓存策略
4. 检查数据库查询效率

### 4.2 安全隐患 (security_vulnerability)

**检查方法:**
1. 搜索硬编码密钥、密码、token
2. 检查 SQL 注入风险
3. 检查 XSS 风险
4. 检查权限控制和输入验证

**注意: 任何安全隐患直接触发 L4 紧急等级**

### 4.3 可维护性 (maintainability)

**检查方法:**
1. 计算圈复杂度（函数嵌套深度、条件分支数量）
2. 检查函数长度（超过 50 行标记）
3. 检查文件长度（超过 300 行标记）
4. 评估模块间的耦合度

### 4.4 可扩展性 (scalability)

**检查方法:**
1. 检查是否有硬编码的限制
2. 评估配置外部化程度
3. 检查是否有扩展点设计
4. 评估状态管理方式

---

## 五、综合评分计算

```
综合健康指数 = 架构健康度 * 0.30
             + 代码质量 * 0.25
             + 流程规范性 * 0.20
             + 非功能性需求 * 0.25
```

趋势计算: 与上次同维度分数对比，得出 ↑(上升) ↓(下降) →(持平)
