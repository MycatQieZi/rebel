# 改朝换代执行手册 (Rebuild Playbook)

本文件定义了改朝换代从分析到执行的完整流程和最佳实践。

## 一、改朝换代概览

改朝换代是叛逆者系统中**最高级别的主动操作**，区别于政变的"修复"思维，改朝换代是从目标出发的"重建"思维。

| 维度 | 政变 (Coup) | 改朝换代 (Rebuild) |
|------|------------|-------------------|
| 触发 | 自动(L3/L4)或手动 | **纯手动** '--rebuild' |
| 视角 | 从问题出发 "修复" | 从目标出发 "重建" |
| 范围 | 局部 | **全局** |
| 深度 | 诊断+调整 | 技术选型+路线图 |
| 执行后 | 更新分数 | **需要重新 /rebel** |

## 二、触发时机

### 适合发起改朝换代的场景

- 技术栈出现多个过时/不再维护的依赖
- 项目的核心目标与当前实现严重脱节
- 功能膨胀——大量功能无人使用或功能重叠
- 架构腐化到"修补"成本高于"重建"成本
- 团队希望系统性升级技术路线
- 项目需要从一个阶段（如原型）过渡到另一个阶段（如生产级）

### 不适合的场景

- 项目刚启动，尚无足够积累
- 正在进行紧急线上修复
- 只有一两个局部问题需要修复（应该用政变）
- 没有足够时间进行系统性重构

## 三、Phase 1 操作指南

### 3.1 目标重塑 (Goal Re-establishment)

**执行步骤:**

1. 从 README.md、AGENTS.md、项目记忆 中提取项目核心目标
2. 扫描当前代码结构，列出所有模块/功能
3. 将每个模块/功能与核心目标对照，计算匹配度
4. 对不匹配的功能，分析原因:
   - "扩展需求" — 核心功能衍生，可保留但需明确归属
   - "方向偏移" — 与核心目标无关，建议移除或独立
   - "临时方案" — 为某次需求临时添加，应清理
5. 输出建议：重新聚焦现有目标 或 更新目标声明

**评分标准:**

- 90%+ 匹配: 目标明确，执行良好
- 70-89% 匹配: 基本聚焦，有少量扩展
- 50-69% 匹配: 偏移明显，需要调整
- <50% 匹配: 严重偏移，建议重新定义目标

### 3.2 技术选型审计 (Tech Stack Audit)

**审计类别:**

| 类别 | 权重 | 检查内容 |
|------|------|----------|
| 编程语言 | 25% | 版本是否过时、社区活跃度、团队熟悉度 |
| 核心框架 | 25% | 维护状态、性能、安全性、生态 |
| 数据库/存储 | 20% | 数据模型匹配度、扩展性、运维成本 |
| 基础设施 | 15% | 部署方式、CI/CD、监控、日志 |
| 关键依赖 | 15% | 是否有替代、安全状态、授权合规 |

**每项技术的评估矩阵:**

- 适配度 (1-10): 技术是否满足项目需求
- 维护状态: 活跃/维护中/仅修复/停止维护
- 迁移成本: low(天)/medium(周)/high(月)
- 社区支持: 强/中/弱
- 安全状态: 良好/有已知问题需要关注/有严重漏洞

**建议分类:**

- **保留**: 适配度高，维护活跃，无需变更
- **升级**: 当前版本过低，建议升级到最新稳定版
- **替换**: 存在更好的替代方案，建议迁移
- **移除**: 不再需要，可安全移除

### 3.3 功能累积审计 (Feature Audit)

**审计方法:**

1. 扫描所有入口点（API路由、CLI命令、事件处理等）
2. 追踪每个功能的调用链，判断实际使用情况
3. 检查是否有功能被注释掉或标记为 DEPRECATED
4. 识别功能重叠——多个模块实现相似功能
5. 分类每个功能

**四分类决策树:**

```
功能A是否还在使用？
├── 是 → 实现质量如何？
│        ├── 良好 → ✅ 保留
│        └── 欠佳 → 🔄 重构
└── 否 → 是否有类似功能？
         ├── 是 → 🔀 合并
         └── 否 → 🗑️ 废弃
```

**分类时必须回答:**

- 为什么分类为此类？
- 如果保留/重构/废弃/合并，对项目有何影响？
- 估计工作量（人天）

### 3.4 重构路线图 (Rebuild Roadmap)

**路线图设计原则:**

1. **渐进式重构**: 每阶段可独立交付，不造成功能中断
2. **风险隔离**: 高风险变更放在早期处理，留足缓冲
3. **依赖优先**: 先处理基础层（被依赖最多的模块）
4. **快速见效**: 第一阶段应包含能快速验证价值的任务

**每阶段必须包含:**

- 阶段目标和可交付物
- 任务清单（每项有负责人/预估工时）
- 前置依赖（需要先完成什么）
- 风险清单和缓解措施
- 验收条件（如何判断完成）
- 回滚方案（失败如何回退）

**时间估算指南:**

- 小型重构（单模块）: 1-3 天
- 中型重构（多模块）: 1-2 周
- 大型重构（全项目）: 2-6 周
- 改朝换代级（包含技术栈迁移）: 1-3 月

## 四、Phase 2 操作指南

### 4.1 文件级变更清单

```
模块: [模块名]
阶段: Phase X

新建文件:
  - path/to/new/file.ts  # 用途说明

修改文件:
  - path/to/existing.ts  # 变更说明

删除文件:
  - path/to/deprecated.ts  # 删除原因（已由 X 替代）

移动文件:
  - path/to/old.ts → path/to/new.ts  # 移动原因
```

### 4.2 接口迁移对照表

```
旧接口                          → 新接口                          兼容期
GET /api/v1/users               → GET /api/v2/users               保持3个月
POST /api/v1/users              → POST /api/v2/users              保持3个月
GET /api/v1/users/:id/posts     → GET /api/v2/users/:id/posts     保持3个月
```

### 4.3 数据迁移方案

对于涉及数据库变更的重构:
1. 记录当前 schema
2. 定义目标 schema
3. 编写迁移脚本（含回滚脚本）
4. 制定迁移策略（在线迁移/停机迁移/双写）
5. 数据完整性校验方案

### 4.4 测试策略

- **回归测试范围**: 列出需要重新测试的功能和场景
- **新增测试要求**: 重构新增的功能需要哪些测试
- **性能测试**: 如果有性能相关变更
- **集成测试**: 端到端测试关键流程

## 五、审批与执行

### 审批流程

Phase 1 输出四份子报告，用户可:
1. **全部批准**: 进入 Phase 2，展开所有阶段的战术计划
2. **选择性批准**: 仅展开已批准部分的战术计划
3. **修改后批准**: 修改路线图后再推进
4. **全部否决**: 终止改朝换代流程

### 执行注意事项

1. 每阶段开始前确认环境可用
2. 每项变更完成后立即验证
3. 保持 commit 粒度，每项变更单独提交
4. 遇到阻塞立即汇报，不硬推
5. 每阶段完成后执行 `/rebel` 评估进度

## 六、改朝换代完成检查清单

- [ ] 所有 Phase 任务按计划完成
- [ ] 所有测试通过（回归 + 新增）
- [ ] 废弃代码已清除
- [ ] 技术栈升级到目标版本
- [ ] 文档已更新（README、API 文档）
- [ ] 已执行 `/rebel` 标准审视，健康评分稳定
- [ ] 相关记忆已更新
- [ ] 经验教训已记录到审视历史
