# 叛逆者政变执行手册 (Coup Playbook)

本文件定义了政变从提案到执行的完整流程和最佳实践。

## 一、政变触发条件

### 自动触发 (危机等级 L3+)

| 条件 | 触发维度 | 紧急度 |
|------|----------|--------|
| 架构健康度 < 40 | 架构 | 高 |
| 代码质量 < 40 | 代码 | 高 |
| 安全漏洞发现 | 非功能性 | 紧急 |
| 目标严重偏移（>60%任务偏离） | 架构 | 高 |
| 多维度同时 < 60 | 跨维度 | 高 |
| 核心功能不可用 | 代码 | 紧急 |

### 手动触发

- 用户执行 `/rebel --coup`
- 用户明确表示"需要重新审视方向"

### 模块级政变触发 (v2.0 新增)

模块政变是 v2.0 的核心变革——每个模块可以独立触发政变，不影响其他模块。

| 条件 | 触发级别 | 行为 |
|------|----------|------|
| 模块任一基础维度 < 40 | L3 | 生成模块政变提案 |
| 模块专属维度 < 30 | L3 | 生成模块政变提案 |
| 模块内安全漏洞 | L4 | 紧急模块政变 |
| ≥2个模块同时 L3+ | — | 自动升级为全局政变 |

模块政变与全局政变的区别：
- **模块政变**：提案范围限定在模块目录内 + 级联影响报告
- **全局政变**：提案可跨模块，统筹多个模块的问题
- **级联审视**：模块政变自动触发依赖该模块的其他模块审视，生成级联影响分析

## 二、政变提案编写指南

### 诊断部分 (Diagnosis)

每个诊断项必须包含:

1. **维度**: 明确指出问题所属维度
2. **严重度**: high 或 critical（L3/L4 级别才进入政变）
3. **描述**: 用具体的技术语言描述问题，避免模糊表述
4. **证据**: 列出 2-5 条具体证据（文件路径、代码行号、数据指标）

好的诊断:
```yaml
- dimension: "architecture_health"
  severity: "critical"
  description: "项目核心目标在过去15次交互中发生严重偏移，从'构建实时聊天系统'偏移到'构建通用工具库'，60%的近期提交与核心目标无关"
  evidence:
    - "commit abc123: 新增通用JSON解析工具，与聊天系统无关"
    - "commit def456: 重构了日志模块，但聊天核心功能仍有3个已知bug未修复"
    - "src/utils/ 目录下7个文件中有5个是通用工具而非聊天相关"
```

差的诊断:
```yaml
- dimension: "architecture_health"
  severity: "high"
  description: "项目方向可能有些偏移"
  evidence:
    - "感觉最近做的事情不太对"
```

### 方案部分 (Proposals)

每个提案必须满足:

1. **类型明确**: 属于5种政变类型之一
2. **行动具体**: 每个 action 都可直接执行
3. **优先级排序**: critical > high > medium > low
4. **影响可评估**: 说明预期效果

### 提案分类规则

| 类型 | 何时使用 | 典型行动 |
|------|----------|----------|
| goal_realignment | 目标偏移严重 | 更新项目记忆中的目标声明，标记偏离任务 |
| memory_repair | 记忆过时/矛盾 | 删除过时记忆，更新错误记忆，创建缺失记忆 |
| architecture_adjustment | 架构腐化 | 重构模块边界，调整依赖关系，提取公共逻辑 |
| process_correction | 规范被忽视 | 恢复 linting 规则，重建 CI 检查，更新规范文档 |
| tech_debt_clearance | 债务积累过多 | 处理 FIXME，替换临时方案，补充测试 |

## 三、提案展示策略

### 展示顺序

1. 先展示**问题全景**（让用户感受到紧迫性）
2. 再展示**具体诊断**（用证据说服）
3. 然后展示**提案方案**（给出解决路径）
4. 最后展示**影响评估**（帮助用户决策）

### 展示原则

- **数据先行**: 每个结论都有数据支撑
- **对比鲜明**: 展示当前状态 vs 理想状态
- **紧迫但不恐慌**: 客观呈现问题，不夸大也不淡化
- **选择权归用户**: 提供选项，但不替用户做决定

## 四、政变执行检查清单

### 执行前:

- [ ] 用户已明确批准提案（全部或部分）
- [ ] 理解每个被批准提案的具体行动
- [ ] 确认执行顺序（按优先级）
- [ ] 备份当前状态（确保可回滚）

### 执行中:

- [ ] 按提案逐项执行
- [ ] 每完成一项，验证变更效果
- [ ] 保持变更最小化（不做超出提案范围的修改）
- [ ] 实时更新执行进度

### 执行后:

- [ ] 验证所有变更按预期工作
- [ ] 更新相关记忆和配置
- [ ] 记录政变结果（成功/部分成功/失败）
- [ ] 生成政变复盘报告

## 六、常见政变场景与最佳实践

### 场景 1: 目标偏移政变

**典型症状:**
- 项目 README 描述的功能与实际代码不符
- 最近多次提交都在做与核心目标无关的事
- 用户对项目的描述已经发生变化

**最佳实践:**
1. 先回顾项目最初的目标声明
2. 列出偏移的具体证据
3. 提出新的目标声明（或恢复原始目标）
4. 标记需要清理的偏离代码

### 场景 2: 技术债务清算政变

**典型症状:**
- TODO/FIXME 数量超过 20
- 多个 HACK/WORKAROUND 标记
- 新功能开发越来越慢

**最佳实践:**
1. 按影响面和紧急度排序债务
2. 分批处理，不要一次性重构所有
3. 每批处理后运行测试确认无回归
4. 建立防止债务再次积累的机制

### 场景 3: 记忆修复政变

**典型症状:**
- Agent 基于过时记忆做出错误决策
- 多条记忆相互矛盾
- 记忆描述了不存在的功能

**最佳实践:**
1. 列出所有可疑记忆
2. 逐条与实际代码交叉验证
3. 标记需要更新/删除/创建的记忆
4. 按影响面排序处理顺序
